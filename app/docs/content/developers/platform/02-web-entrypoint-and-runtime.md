## Overview

When Spacewave runs in the browser, the entrypoint script bootstraps a multi-worker architecture: a SharedWorker runs the Go WASM runtime, a ServiceWorker intercepts network requests, and the main thread renders the React UI. The entrypoint is generated by bldr from the project config and handles initialization, worker registration, and root component mounting.

## Entrypoint Script

The entrypoint (`bldr/web/entrypoint/entrypoint.tsx`) initializes React and the bldr runtime. At build time, bldr injects several constants:

| Constant | Purpose |
|----------|---------|
| `BLDR_RUNTIME_JS` | Path to the Go WASM runtime worker script |
| `BLDR_SW_JS` | Path to the ServiceWorker script |
| `BLDR_SHW_JS` | Path to the SharedWorker script |
| `BLDR_STARTUP_JS` | Path to the app startup component |
| `BLDR_FORCE_DEDICATED_WORKERS` | Forces dedicated workers instead of shared |

These paths are resolved at build time and point to the correct hashed filenames in the dist bundle.

## Worker Architecture

The browser runtime uses three worker types:

**SharedWorker** - Runs the Go WASM runtime and plugin host. Shared across all tabs for the same origin. Handles block storage, world state, RPC routing, and plugin execution. The unified worker supports both native and QuickJS plugins via URL parameters.

**ServiceWorker** - Intercepts fetch requests and routes them to the Go runtime for processing. This enables the Go code to serve HTTP responses (plugin assets, API endpoints) without a network round-trip.

**Main Thread** - Renders the React UI and communicates with the SharedWorker via starpc over MessagePort. UI components call SDK RPCs that are routed to the Go runtime in the SharedWorker.

If SharedWorker is unavailable (some browser configurations), the runtime falls back to a dedicated Worker with `forceDedicatedWorkers`.

## Startup Component

When `load_web_startup` is set in the project's `StartConfig`, the entrypoint lazy-loads the specified React component and renders it as the children of `BldrRoot`. The startup component typically contains a `<WebView />` that registers with the Go runtime and receives render mode instructions.

```typescript
// The startup component is lazy-loaded:
const LoadedComponent = React.lazy(
  async () => import(BLDR_STARTUP_JS),
)
```

The startup component is wrapped in `Suspense` with a loading fallback and `WebViewErrorBoundary` for error isolation. When a startup component is provided, the root WebView is disabled (`disableRootWebView: true`) since the startup component manages its own WebView.

## Hydration

If the HTML container has a `data-prerendered` attribute (from the static prerender build), the entrypoint uses `hydrateRoot()` instead of `createRoot()`. This preserves the server-rendered HTML and attaches event handlers without a full re-render. The attribute is removed after hydration to prevent double-hydration on subsequent navigations.

## WebDocument Options

The entrypoint constructs a `WebDocumentOptions` object from the injected constants and URL parameters:

- `runtimeWorkerPath` - Where to find the Go WASM runtime worker.
- `serviceWorkerPath` - Where to find the ServiceWorker.
- `sharedWorkerPath` - Where to find the SharedWorker.
- `webDocumentId` - From the `webDocumentId` URL parameter (used by Electron for multi-window support).

These options are passed to `BldrRoot`, which initializes the worker infrastructure and establishes the RPC channel.
